## 🔴 必改（安全 / 會出大事）

### A1. JWT 存在 localStorage → XSS 可竊取（安全｜脆弱）

- 位置：第 3 節步驟 6。
- 問題：localStorage 可被任何 JS 讀取，一旦頁面有 XSS，token 直接被偷。
- 修正：改存 **httpOnly + Secure + SameSite=Lax/Strict** 的 cookie，前端拿不到也偷不走。

### A2. OAuth 流程缺 `state` 參數 → CSRF（安全｜邊緣/依賴）

- 位置：第 3 節整個 callback 流程。
- 問題：沒有 `state` 參數做 CSRF 防護，攻擊者可偽造 callback 把受害者登入成攻擊者帳號（login CSRF）。
- 修正：發起授權時帶隨機 `state`，callback 驗證一致；建議加 PKCE。

### A3. 密鑰寫死在程式碼常數（安全｜未達標）

- 位置：第 6 節。
- 問題：直接違反 `REQUIREMENTS.md`「不可把密鑰寫死」。一旦進 git 就外洩，且無法分環境。
- 修正：放環境變數 / secret manager，啟動時檢查缺漏。

### A4. 沒有 token 撤銷 / 真正的登出（安全｜脆弱）

- 位置：第 5 節。
- 問題：JWT 無狀態，刪 localStorage 只是「本機忘記」，token 在 24h 內**仍然有效**。被偷的 token 無法主動失效。
- 修正：導入 refresh token + 短期 access token、或伺服器端 session/黑名單（需共享儲存）。

## 🔴 必改（依賴 / 維運 / 需求未達標）

### A5. migration 無 rollback，且叫人在正式庫直接執行（依賴｜未達標）

- 位置：第 2 節（「直接在正式資料庫執行」）。
- 問題：違反「失敗可回滾」需求，也違反本書「為了驗證而計畫」。直接動正式庫沒有退路。
- 修正：用 migration 工具（含 up/down），先在測試環境驗證，備好 rollback。

### A6. 頻率限制存在單程序記憶體 → 多實例失效（依賴｜脆弱）

- 位置：第 4 節（`const counts = {}`）。
- 問題：需求明說會水平擴展。記憶體計數器在多副本下各算各的，限制形同虛設；重啟即歸零。
- 修正：用 **Redis** 之類的共享儲存做 rate limit。

### A7. 同 email 跨 provider 帳號衝突未處理（邊緣｜未達標）

- 位置：第 2 節 schema + 第 3 節步驟 4。
- 問題：需求明說有人 Google/GitHub 同 email 要視為同一帳號。但目前用 email 找/建，schema 也沒 unique 約束，會產生重複帳號或誰先誰贏的混亂。
- 修正：設計帳號連結（identities 表，user 1:N provider），email 正規化，明確衝突策略。

## 🟡 建議（會痛但不致命）

### A8. 沒有 refresh token / 24h 到期後的 UX（邊緣｜脆弱）

- 位置：第 3 節步驟 5。
- 問題：access token 一到 24h 直接過期，使用者操作到一半被踢登出，無續期。
- 修正：短期 access + 長期 refresh，到期自動續，或滑動過期。

### A9. 外部 OAuth provider 失效無處理（依賴｜未達標）

- 位置：第 3 節（拿 code 換 token 那步）。
- 問題：需求要求 provider 連不上時要友善錯誤。計畫完全沒提逾時/重試/錯誤頁。
- 修正：加 timeout、重試、降級錯誤畫面，不要噴 500/白畫面。

### A10. E2E 測試只有 happy path（邊緣｜未達標）

- 位置：第 8 節。
- 問題：需求要 Playwright 測試，但只測成功登入。失敗案例（使用者拒絕授權、過期/竄改 token、超過頻率限制、provider 逾時）全沒測。
- 修正：補失敗與邊緣案例，把「完成定義」綁上這些測試。

### A11. 任務排序錯誤 / 假平行（依賴｜脆弱）

- 位置：第 7 節任務表。
- 問題：T3 前端「串接 API」卻標成與 T2 後端可平行——前端依賴 T2 的 API 合約，硬平行會卡住或重工。測試 T6 排最後也違反驗證優先精神。
- 修正：先定 API 合約再平行；測試應隨功能同步、甚至先行（TDD）。

## 復盤：這題想教你的

1. **大多數致命問題是「沒寫出來的東西」**：缺 `state`、缺 rollback、缺失敗測試——審查要找的常是**缺席**而非錯字。
2. **拿需求逐條對照**：A3/A5/A7/A9/A10 都是「需求白紙黑字寫了、計畫卻沒做」，對照清單就能抓。
3. **問三個魔法問題**：①惡意使用者會怎麼玩？②這東西跑多份會怎樣？③失敗時怎麼退？——A2/A6/A4/A5 全是這三問逼出來的。
4. **這就是 80/20 的價值**：以上 11 點若等寫完幾千行才發現，等於整個 auth 模組重來；在這份純文字計畫改掉，成本趨近於零。
