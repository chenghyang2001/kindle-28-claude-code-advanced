# 原始需求：使用者認證系統（OAuth 登入）

> 這是「業主」交付的需求。審查員（Session B）要用這份來檢查 `PLAN.md` 有沒有達標、有沒有過度設計、有沒有漏掉。

## 目標

為一個既有的 Web 應用（Next.js + PostgreSQL）加入第三方登入，讓使用者能用社群帳號登入，不必另記密碼。

## 功能需求

1. 支援 **Google** 與 **GitHub** 兩種 OAuth 登入。
2. 登入後以 **JWT** 維持登入狀態，**24 小時** 過期。
3. 對登入相關端點做 **頻率限制**：每分鐘最多 5 次嘗試。
4. 需有 **登出** 功能。
5. 使用者基本資料（email、顯示名稱、頭像、來源 provider）存入資料庫。
6. 需有 **Playwright** 寫的 E2E 測試。

## 非功能需求

- **安全**：不可把任何密鑰（OAuth client secret、JWT signing key）寫死在程式碼裡。
- **可維運**：資料庫變更要能安全套用，且**失敗可回滾**。
- **可用性**：第三方服務（Google/GitHub）暫時無法連線時，使用者要看到友善錯誤，而不是白畫面或 500。
- **多實例**：服務會跑多個副本（水平擴展），任何「狀態」都不能只存在單一程序的記憶體裡。

## 明確邊界（Out of Scope）

- 不做傳統 email + 密碼註冊。
- 不做雙因素驗證（2FA）。
- 不做後台管理介面。

## 已知情境（業主補充，容易被忽略）

- 有些使用者的 **Google 與 GitHub 綁的是同一個 email**，他們預期登入後是「同一個帳號」，不是兩個。
- 公司內部測試者常**多人共用同一個對外 IP**（NAT）。
- 行銷活動時登入流量會在短時間暴衝。
