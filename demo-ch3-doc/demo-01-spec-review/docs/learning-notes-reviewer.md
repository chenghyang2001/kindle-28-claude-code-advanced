## 漏掉的那一條：A8

**A8 = access token 到 24h 直接過期，使用者操作到一半被踢登出、無 refresh / 滑動續期（UX/可用性）。**

### 根因（為什麼會漏）

1. **鎖在「安全獵捕模式」**：同一段「登出」區塊藏了兩個坑——A4（被偷的 token 還有效，**安全**角度）與 A8（正常人被突然登出，**UX**角度）。火力全開在凶的 A4，放過了煩的 A8。
2. **把需求照單全收、沒追問體驗**：看到「JWT 24 小時過期」就當成不可動的鐵律，沒問「過期那一瞬間，正常使用者的體驗是什麼？」A8 要的不是改 24h，是補 refresh。
3. **對「缺席」反應慢**：A8 是「沒寫出來的東西」。答案卷復盤第 1 點：**審查要找的常是缺席，不是錯字**。我對「缺 refresh」這個缺席慢了一輪——到 v2 規劃者主動端出 refresh，我才接話講得頭頭是道（R3 絕對上限、R4 輪替）。

## 固化成 checklist：四個受害者視角

審查同一段設計/程式時，**刻意切換四種視角**各跑一輪，避免只跑其中一種就收手：

| # | 視角 | 要問的問題 | 這次對應的坑 |
|---|---|---|---|
| 1 | 😈 惡意使用者 | 攻擊者會怎麼玩？能偷/偽造/越權嗎？ | A2 缺 state、A4 token 不能撤銷、A1 localStorage 被偷 |
| 2 | 👥 跑多份 | 這東西水平擴展、跑多副本會怎樣？ | A6 rate limit 記憶體各算各的 |
| 3 | 💥 失敗怎麼退 | 外部服務掛了、要回滾時怎麼辦？ | A5 migration 無 rollback、A9 provider 失效無友善錯誤 |
| 4 | 🙂 正常人用起來爽不爽 | happy path 走完後，正常使用者的體驗順嗎？ | **A8 24h 被突然登出（← 這次漏的就是少跑了這一輪）** |

> 答案卷原本給「三個魔法問題」（①惡意 ②多份 ③失敗）。實戰證明要**擴成四個**，補上「④正常人體驗」——A8 正是只有第 4 視角才逼得出來。

---

## 一句話帶走

> 致命的問題多半擋得住（7 個 🔴 全中），真正會溜走的是「不致命但煩」的 UX/可用性缺席。
> 解法不是更用力挑錯，而是**換視角**——尤其別忘了那個最容易被忽略的「正常使用者」。
