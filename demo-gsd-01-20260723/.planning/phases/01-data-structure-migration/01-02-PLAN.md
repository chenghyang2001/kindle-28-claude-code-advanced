注意：呼叫端傳入的 name 仍是 str，不要改成 Task。函式簽名 delete_task(tasks, name: str) 不變。

<threat_model>
本 phase 為本機 in-memory CLI，無網路 / DB / 認證面。本 plan 的威脅面集中在 delete_task 重寫的正確性與資料完整性。

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| name 字串 → delete_task 比對邏輯 | 使用者提供的名稱字串用於定位並移除清單元素；比對邏輯若靜默失效會導致「以為刪掉、其實沒刪」的資料完整性錯覺 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| ----------- | ---------- | ----------- | ------------- | ----------------- |
| T-01-04 | Tampering (完整性) | delete_task 換 Task 後 name in tasks 靜默失效 | mitigate | 整段重寫為 `for t in tasks: if t.name == name` 顯式比對；以 test_delete_existing_task_returns_true 與 CLI 煙霧測試雙重驗證「確實刪除」而非靜默 no-op |
| T-01-05 | Tampering | 誤用「全部同名一起刪」寫法破壞只刪第一筆語意 | mitigate | 明令採迴圈 + 命中即 return，保留 test_delete_only_first_occurrence 作為守門測試 |
| T-01-06 | Supply chain | 相依套件 | accept | 本 plan 無任何新套件安裝，僅 stdlib 與既有 pytest，無 legitimacy gate 需求 |
</threat_model>
