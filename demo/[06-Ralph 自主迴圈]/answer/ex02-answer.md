# 練習 02 解答 — 為重複任務設計安全自主迴圈

## 任務 1：YouTube 夜間爬蟲迴圈的每輪 I/O

- **迴圈單位**：一支影片，逐支處理。
- **每輪輸入**：① 待處理清單取一支影片 URL＋字幕；② 讀狀態檔（哪些已處理）。
- **每輪輸出**：① NotebookLM notebook ＋ 4 種 studio；② Gmail/Telegram 通知；③ **狀態檔標記此影片完成**。
- **收斂**：待處理清單清空。
- **關鍵**：狀態檔讓迴圈**冪等可續跑**——斷掉重跑不重做、不漏做。

## 任務 2：5 個護欄（各擋一種失敗）

| # | 護欄 | 擋掉什麼失敗 |
|---|------|------------|
| 1 | **冪等狀態檔**（處理完打勾，每輪先檢查跳過已處理） | 重複建 notebook、**重複發通知**（斷線重跑早上收一堆重複訊息） |
| 2 | **單支失敗重試上限**（最多重試 2 次→標 failed 跳過） | 一支壞影片**卡死整個迴圈**，害後面全沒處理 |
| 3 | **總迭代上限 + 時間窗**（最多 20 支 / 90 分鐘） | 訂閱爆量或空轉，**跑到天亮燒光額度** |
| 4 | **外部額度/速率護欄**（呼叫間隔 + 當日上限） | 狂打把帳號額度爆掉、被**限流封鎖** |
| 5 | **每輪記錄進度 + 可回滾**（log / commit / 更新狀態檔） | 中途崩潰後**不知做到哪、無法續跑** |

## 延伸思考：最危險的失敗模式？怎麼在設計時堵住
- **最危險 = 「失控空轉 + 越改越糟」**：迴圈在錯誤狀態下持續行動，每輪把問題放大，且無人察覺（半夜、無監督）。
- **設計時就堵**（而非事後）：
  - **零進展偵測**：本輪 = 上輪就停（堵空轉）。
  - **驗證 gate**：每輪後跑測試/lint，紅燈**停不硬推**（堵越改越糟）。
  - **每輪 commit 檢查點**：可回滾到任一輪（壞了能退）。
  - **硬上限（迭代/時間/成本）**：最終保命煞車。
- 一句話：護欄不是事後的 except，是設計時先問「**半夜會怎麼壞**」再逐條堵住。
