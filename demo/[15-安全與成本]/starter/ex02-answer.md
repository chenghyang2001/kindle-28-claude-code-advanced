# 練習 02 解答 — 系統化除錯定位一個 bug

## 核心精神
除錯不要用猜的，用系統化方法：二分搜尋 + 假設-驗證。

## 任務 1：二分搜尋（git bisect）

```bash
git bisect start
git bisect bad                  # 目前壞的
git bisect good <舊的好 commit>  # 已知好的
# git 自動 checkout 中間點 → 測 → git bisect good/bad，反覆二分
git bisect reset
```

**100 個 commit 約 7 次抓到兇手**（`100→50→25→12→6→3→2→1`，log₂100≈7）。

| 方法 | 100 | 1000 | 10000 |
|------|-----|------|-------|
| 逐一試 | 100 次 | 1000 | 10000 |
| 二分 | 7 次 | 10 | 14 |

> 直覺：每次測試帶最大資訊量——逐一試每次排除 1 個，對半砍每次排除一半。手動除錯同理：可疑碼對半註解縮小。

## 任務 2：根因與修法（範例格式）
- 為什麼壞：定位到某 commit 把 `timeout` 從 30 改成 5，導致慢請求被中斷。
- 怎麼改：改回合理值 + 加重試。
- 如何防再犯：補一個整合測試覆蓋慢請求情境 + 在設定值旁註記「為何是這個數」。

## 假設-驗證：為什麼要「最小實驗」
- 一次只改跟假設有關的一個變數，其他不動 → 得到乾淨因果「就是這個原因」。
- 一次改一大堆 → bug 消失也不知是哪個修的（得到結果卻得不到知識），混入新問題也分不清。
- = 第 11 課「小步前進」同理：把變因隔離到最小。

## 延伸思考：理由化行為與預防
- 理由化＝遇困難找藉口偷懶：「檔案太大」(懶讀)、「測試太複雜」(不想修)、「需超範圍」(逃避)、「擔心副作用」(不想動)。
- 預防：① prompt 事先堵死藉口（用工具讀完範圍、遇錯先分析根因）② 要求卡住也回報「已嘗試步驟 + 卡在哪」。
- = agent-rules 反理由化對照表：每種藉口對應一個正確動作。

## 關鍵收穫
- 二分搜尋以對數級速度縮小範圍，每步排除一半。
- 假設-驗證一次只動一個變數，得到乾淨因果；反理由化逼 AI 執行不找藉口。
