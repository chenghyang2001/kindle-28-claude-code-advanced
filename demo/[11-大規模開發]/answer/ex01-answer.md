# 練習 01 解答 — 為模組規劃測試金字塔

## 測試金字塔三層（由下往上，數量遞減）

| 層 | 測什麼 | 速度/穩定 | 數量 |
|----|--------|----------|------|
| 🟩 單元測試 | 最小單位：純函式、驗證邏輯，不碰 DB/網路 | 毫秒級、快又穩 | 最多 |
| 🟨 整合測試 | 多元件接縫：查 DB、呼叫 API | 較慢（真連 DB/起服務） | 適量 |
| 🟥 端對端 E2E | 模擬真實使用者全流程（註冊→登入→下單） | 最慢、最脆弱 | 最少 |

## 為什麼是金字塔（下寬上窄）
- 單元測試又快又穩又便宜 → 寫幾百個、每次存檔都跑。
- E2E 又慢又脆弱又貴 → 只留給最關鍵主流程。
- 倒三角形（大量 E2E）會：測試超慢、常無故失敗 → 大家不跑 → 測試形同虛設。
- 形狀＝把資源花在刀口上。

## 任務 1：某中型模組的金字塔（範例）
- 單元層：金額解析、欄位驗證、純計算函式各自的 happy/edge/error。
- 整合層：service 查 DB 回傳正確、呼叫外部 API 的錯誤處理。
- E2E 層：一條關鍵主流程（例如建立訂單→查詢）。

## 任務 2：覆蓋缺口（至少 2 個）
1. 錯誤處理分支：只測快樂路徑，`except`/`else` 從沒被跑到。
2. 邊界條件：空值、零、極大值未測。
（用 `pytest --cov=mypkg --cov-report=term-missing` 找出 missing 行號。）

## 關鍵收穫
- 三層比例由下往上遞減；用最便宜的測試擋掉大部分問題。
- 覆蓋缺口最常躲在「錯誤分支」與「邊界條件」。
