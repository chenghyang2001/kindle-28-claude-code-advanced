# 練習 02 解答 — 對 repo 做一次小型 audit

## 找風險的三大面向

| 面向 | 嚴重度 | 具體要找什麼 |
|------|--------|-------------|
| 🛡️ 安全 | 最致命（外洩/被駭） | 硬編碼機密、未參數化 SQL（injection）、缺輸入驗證 |
| 🐛 正確性 | 默默做錯事 | 被吞掉的例外（空 catch）、未處理的錯誤路徑 |
| 🔧 維護性 | 慢性病、越來越難改 | 超長函式、重複邏輯、缺測試 |

> 這三類是第 9 課 code-review 三類的放大版：code-review 看「這次改動」，audit 看「整個專案」。

## 任務 1：audit 發現（範例格式，至少 5 項、含 檔案:行號 + 分級）

| 分級 | 發現 | 檔案:行號 | 為什麼是問題 | 建議修法 |
|------|------|-----------|-------------|---------|
| 高 | DB 密碼硬編碼 | config.py:12 | 機密入庫，外洩即被駭 | 改讀 `os.environ` + 進 .gitignore |
| 高 | 字串組合 SQL | db.py:48 | SQL injection 風險 | 改參數化查詢 |
| 中 | 空 except 吞例外 | worker.py:90 | 錯誤被吞，難排查 | 捕捉具體例外 + logging |
| 中 | 函式 200+ 行 | report.py:30 | 難讀難測難改 | 提取子函式 |
| 低 | 重複的 auth 邏輯 | a.py / b.py | 改一處漏一處 | 抽共用模組 |

## 任務 2：最嚴重項的修補方向
- 取「DB 密碼硬編碼」：① 立即把金鑰移出程式碼改環境變數；② 將設定檔加入 .gitignore；③ **rotate 舊金鑰**（已外洩到 git 歷史的視同作廢）；④ 補一個啟動時檢查必要環境變數的 guard。

## 為什麼一定要分級
資源有限、不可能一次修完。50 項平等攤平 → 接手者傻眼、最後一個都沒修。分高/中/低 → 一眼知道先救高的（會出事的）。分級＝替決策者排「先做什麼」，比找出問題本身更有價值。

## 延伸思考：先讀什麼最省時間
目錄結構 → git log（近期活躍處）→ 進入點順資料流。順序重要，因為先有全局地圖才不會在細節裡迷路。

## 關鍵收穫
- 風險分安全/正確性/維護性三面向；安全最優先。
- 每個發現附 檔案:行號 + 原因 + 修法；並用高/中/低分級排優先序。
