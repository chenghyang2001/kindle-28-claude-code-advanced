# 練習 02 解答 — 規劃一次安全重構

## 定義
重構 = 改善內部結構、**不改變對外行為**。使用者看不出差別，程式碼更好讀好改。

## 任務 1：重構步驟與每步驗證（分小步，不一口氣大改）

```text
1. 重構前：確認/補 characterization test，釘住現有行為，測試全綠   ← 起點
2. 小步改：每次只處理一個關注點（如拆一個長函式），改完馬上跑測試保持綠
3. 每個綠燈檢查點 → git commit（沿路放可回滾的存檔點）
4. 全改完：對照對外輸出，與重構前一模一樣
```

**為什麼分小步 + 頻繁 commit**：改壞時損失只有「上個 commit 到現在」，一回滾即安全；一口氣改 300 行才紅燈 → 不知哪步壞、只能重來。本質＝把除錯範圍縮到最小。

## 任務 2：用測試當安全網
安全網 = 兩樣組合：
| 組成 | 保證 |
|------|------|
| characterization test | 行為變了**馬上知道** |
| 頻繁 git commit | 變壞了**馬上退回去** |

**必須事前備好，不能邊改邊補**——安全網的意義是動手前先確立「正確基準線」；邊改邊補會分不清測的是原行為還是新行為。鐵律：**先張網，再跳。**

## 延伸思考：migration 為何分批
一次到底 = 同時改幾十檔，壞了不知哪個造成、回滾衝擊巨大。分批 = 風險切小塊，每塊可獨立驗證、回滾。

## 關鍵收穫
- 安全重構＝小步 + 每步測試 + 每綠燈 commit。
- 安全網（測試 + commit）必須事前備好；先張網再跳。
