# 練習 03 解答 — 分批 migration 計畫（擴展—收斂法）

## 情境
改一個被上百處呼叫的 API 簽名，不能讓專案中途壞掉。

## 擴展—收斂法（expand / contract）三批次

| 批次 | 名稱 | 做什麼 | 此刻系統 |
|------|------|--------|---------|
| 1 | 🔵 擴展 expand | 新增新簽名；舊簽名保留為 deprecated wrapper（內部轉呼叫新函式） | 新舊並存，完全沒壞 |
| 2 | 🟡 遷移（中間步） | 逐一把呼叫點改到新簽名，**一檔一 commit**（可單獨驗證/回滾） | 一檔一檔安全搬 |
| 3 | 🟢 收斂 contract | 所有呼叫點改完，才刪掉舊 wrapper | 回到「只有一套」乾淨狀態 |

**精髓**：先擴張 → 容忍新舊並存 → 再收合。永遠不讓系統處在「壞掉」中間狀態，每批次結束都能正常跑。
批次 2 ＝練習 02「小步 + 頻繁 commit」放大到整個專案。

## 限制條件達成
- ✅ 可逐批驗證與回滾：每批次結束系統可跑、每檔一 commit。
- ✅ 客觀「全部完成」驗收條件：
  1. 全部測試綠（行為沒改壞）
  2. `grep` 全專案搜不到舊簽名殘留（工具實掃、不憑感覺）

## 範例驗收指令
```bash
pytest -q
grep -rn "old_signature(" src/   # 應為 0 筆
```

## 關鍵收穫
- expand/contract 用「新舊並存」換取「過程零中斷」。
- 完成定義必須客觀：測試全綠 + grep 不到舊用法。
