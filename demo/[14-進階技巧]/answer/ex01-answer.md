# 練習 01 解答 — 用多步拆解重寫弱 prompt

## 任務 1：弱 prompt 的三個弱點（以「做一個使用者登入系統」為例）

| 弱點 | 問題 |
|------|------|
| 範圍模糊 | 要不要註冊/忘記密碼/記住我/第三方登入？AI 自己猜範圍 |
| 沒技術規格 | 用什麼 DB？密碼怎麼存？session 還是 JWT？AI 亂選或選到不相容 |
| 沒驗收標準 | 怎樣算做完做對？AI 不知做到哪、你也沒法客觀判斷 |

> 根本問題：把所有判斷與猜測丟給 AI 決定，每猜一次多一個「跟你想的不一樣」風險。

## 任務 2：改寫成多步拆解 + 成功標準

**弱：**
```
做一個使用者登入系統
```

**強：**
```
分 4 步做登入：1) 資料表 schema 2) 註冊/登入 API（含錯誤碼）
3) 密碼 bcrypt + JWT 4) 端對端測試（註冊→登入→存取受保護資源）。
先列步驟與每步驗收標準，我確認後再逐步實作。
成功標準：錯誤密碼回 401；token 過期回 401；測試全綠。
```

## 為什麼「先寫成功標準」重要
成功標準＝驗收的共同標準（可客觀檢驗的是非題，非模糊形容詞），解決三件事：
1. AI 有明確目標，朝標準實作、不交半成品。
2. 你有客觀驗收依據，一條條對照、不憑感覺扯皮。
3. 逼你自己先想清楚——寫不出標準＝還沒想清楚，更不該急著開工。

> 呼應 dev-workflow「規格先行」+「複雜需求先拆解、不要 one-shot」。

## 關鍵收穫
- 弱 prompt 弱在範圍模糊/無規格/無驗收，把判斷丟給 AI 猜。
- 多步拆解 + 成功標準＝把判斷權收回自己手上。
