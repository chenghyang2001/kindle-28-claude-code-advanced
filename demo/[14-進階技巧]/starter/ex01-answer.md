# 練習 01 解答 — 設計三 agent 團隊

## 任務 1：三 agent 分工與交付物（以「為模組加新功能」為例）

```text
研究 agent（只讀）  -> 交付「相依地圖 + 風險清單」給主 agent
實作 agent（寫 src）-> 依研究結論實作，交付「程式碼 + 單元測試」
審查 agent（只讀）  -> 交付「必改/建議清單」
主 agent           -> 派工、整合、處理衝突；不下場寫業務碼
```

## 為什麼拆三個而非一條龍
| 好處 | 說明 |
|------|------|
| 乾淨 context | 各自只裝相關資訊，不混淆 |
| 獨立審查 > 自我審查 | 自審有盲點＋「我寫的當然對」偏差；中立審查更抓得到 |
| 專業分工 | 審查可特別多疑、實作可特別專注 |

> 成本：多花 token 換少犯錯，划算（埋 bug 的修復成本遠超多請一審查 agent）。

## 主 agent 為何不下場寫碼
- 指揮 ≠ 執行：價值在綜觀全局；跳下去寫 → 注意力被細節吸走、退化成工人、團隊群龍無首。
- 獨立性：自己寫了碼，整合時會偏袒 → 掉回自我審查盲點。
- 原則：指揮者與工人職責分離（指揮官調度、不親自衝鋒）。

## 任務 2：合約邊界（讀/寫）
- 研究 agent、審查 agent = 唯讀（只看不改）
- 只有實作 agent 能寫 `src/`
- 邊界講死 → 不搶改同檔、不踩線（= 第 9 課 code-review 只讀、simplify 才改同理）

## 關鍵收穫
- 三 agent 拆分的核心好處是乾淨 context + 獨立審查。
- 指揮者不下場、讀寫邊界講死，是團隊不亂的兩大前提。
