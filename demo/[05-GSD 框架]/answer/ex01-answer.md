# 練習 01 解答 — 把長型專案拆成 GSD 階段

## 專案
把 `say.py` ＋ `say_ui.py` 整合重構成一個正式的語音套件（長型專案，適合 GSD）。

## 任務 1 + 任務 2：phases、產出物、gate

| Phase | 階段 | 產出物 | gate（客觀過關條件） |
|-------|------|--------|---------------------|
| 0 | 規格 | 規格文件（對外功能、CLI 介面、import 方式） | 規格經審查通過 + 功能清單/CLI 定案、無待決問題 |
| 1 | 骨架 | 目錄結構 + 函式簽名空殼 + 型別標註 | 能 import 不報錯 + 通過編譯/語法檢查 + 簽名與型別齊全 |
| 2 | 核心實作 | 合成/播放/剪貼簿邏輯 + 各模組單元測試 | **單元測試全綠** + 覆蓋率 ≥ 80% |
| 3 | 介面整合 | CLI + GUI 兩入口接上核心 | 兩入口各跑一次冒煙測試能啟動並出聲、無 crash |
| 4 | 打包與文件 | 打包 exe + README + 範例 | exe 在乾淨環境可跑 + README 照做可成功 + 範例可重現 |

## 重點心得
- 每階段都有**具體交付物**，不是模糊的「進度 70%」。
- 每個 gate 都**客觀可判定**（測試綠 / 可編譯 / 審查通過 / 端對端跑得起來），
  不能是「感覺差不多」「應該沒問題吧」——否則 gate 失去守門意義。

## 驗收
- [x] 至少 3 個階段，每階段有可驗收產出（這裡 5 個）
- [x] 每個 gate 客觀可判定
