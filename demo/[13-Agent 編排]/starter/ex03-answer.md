# 練習 03 解答 — 端到端自動化規格

## 範例任務：YouTube 夜間爬蟲
排程每晚掃訂閱頻道 → 篩相關影片 → 派 subagent 用 skill 丟 NotebookLM 處理 → 發通知。
（組合：排程觸發 + skill 封裝流程 + subagent 平行處理多支影片）

## 三種護欄（各防什麼災難）

| 護欄 | 防什麼 | 怎麼運作 |
|------|--------|---------|
| ⏱️ 逾時上限 timeout | 卡死 | 某步一直不回 → 半夜掛機白燒資源。設時間天花板，超過強制中斷 |
| 📢 失敗通知 | 默默出包沒人知道 | 出錯立刻發 Telegram/Email「掛在第幾步」，讓失敗被看見 |
| 🔒 單例鎖（防重入） | 自己撞自己 | 跑超過排程間隔 → 兩實例同跑搶資源/重複處理。開跑前檢查「是否已有一個我在跑」 |

## 各環節失敗處理（題目限制條件）
- 原則：別讓單環節失敗無聲拖垮整條，也別假裝沒發生。
- 每步明確失敗處理：不可空 try/except 吞錯繼續跑（靜默失敗大忌）。
- 合理降級：5 支影片第 3 支 API 掛 → 標記失敗+記錄+通知，繼續處理第 4、5 支（收穫 4 支只損失 1 支）。
- 例外：失敗後繼續沒意義者（如最初認證就失敗）→ 早早停 + 通知。
- 判斷關鍵：這步失敗後面還有沒有意義？有 → 隔離失敗繼續；沒 → 乾淨停 + 通知。

## 排程護欄範例（cron 概念）
```text
0 2 * * *  cd /path/repo && flock -n /tmp/crawler.lock \
  timeout 1800 claude -p "/youtube-night-crawler" >> log 2>&1 || notify "爬蟲失敗"
# flock=單例鎖, timeout=逾時, || notify=失敗通知
```

## 關鍵收穫
- 無人值守三護欄：逾時防卡死、通知防沉默、單例鎖防重入。
- 自動化的精神是「設好護欄、能安全地失敗並讓你知道」，不是「設好就不管」。
