# 練習 01 解答 — 把多步流程寫成 skill 規格

## skill 規格的兩個關鍵部分

| 部分 | 比喻 | 作用 | 缺了會怎樣 |
|------|------|------|-----------|
| description（含觸發條件） | 🔔 門鈴 | 讓系統判斷「這句話該不該觸發我」 | skill 永遠不被叫出來（更慘） |
| 步驟（SOP） | 📋 流程 | 觸發後照著做什麼 | 被叫出來卻不知幹嘛、空殼 |

> 觸發詞設計往往比步驟更難更重要——門鈴按不到，再好的 SOP 都白搭。

## 觸發詞為何要精準（模糊的兩種錯）
| 錯誤 | 情況 | 後果 |
|------|------|------|
| 漏接 | 話跟觸發詞對不上 | 自動化睡著、還要手動做 |
| 誤觸 | 觸發詞太廣太常見 | 隨口一句就跳出來搶事、打斷對話 |

→ 好觸發詞：涵蓋使用者真會講的各種說法（中英文/同義詞/口語），但不廣到掃進不相關的話；常附「不觸發」反例。

## 任務 1：自訂 skill 範例（以「日報」為例）
```markdown
---
name: daily-report
description: 收集當天 session summary 產出日報。當使用者說「日報」「daily report」「今日總結」時觸發。不觸發：單一 summary、YouTube 摘要。
---
# daily-report
## 步驟
1. 掃描當天 session summary 檔
2. AI 總結成日報（含原文）
3. 輸出 Word + 通知三管道
4. 印出處理的 session 數
```

## 任務 2：三類測試案例
- ✅ 正常：有 3 份 summary → 正確產出含 3 份的日報
- 🔲 邊界：當天 0 份 summary → 不爆，回「今日無 summary」
- 🔴 錯誤：summary 目錄不存在 → 明確報錯，不默默壞掉

## 關鍵收穫
- skill = description（門鈴）+ 步驟（SOP），兩者缺一不可。
- 觸發詞要精準，防漏接與誤觸；三類測試案例齊全。
