# 練習 02 解答 — 用 `@import` 模組化臃腫的 CLAUDE.md（進階）

> 互動三題：拆分依據 → 動手拆 10 條 → 模組化的代價。
> 活教材：使用者自己的 `~/.claude/CLAUDE.md` 就是 `@instructions/*.md` 模組化範本。

---

## 一、為什麼要模組化（一份大 CLAUDE.md 的三病）
1. **太長**：每次開 session 整包灌進 context，吃額度。
2. **難維護**：改一條規則要在幾百行裡撈。
3. **無法跨專案重用**：通用規則與專案規則混在一起，搬不乾淨。

解法：**薄主檔 + 多個主題子檔**，主檔用 `@路徑` 引入。

---

## 二、第一題：拆分的依據

**一句話**：按關注點切——一個子檔只講一個主題，且能**獨立理解、獨立重用**。

| 依據 | 怎麼切 | 範例 |
|------|--------|------|
| ① 按主題，不按長度 | 看講哪件事，非「>50 行就切」 | no-hardcoded-paths／code-quality／dev-workflow／tech-stack |
| ② 按變動頻率 | 常改 vs 很少改分開 | tool-commands/踩坑（常增補）獨立；語言鐵律、唯讀 DB（幾乎不動）放穩定檔 → git diff 乾淨 |
| ③ 按重用範圍 | 通用 vs 專案分開 | 全域 `instructions/*`（跨專案搬） vs 專案 CLAUDE.md（只此 repo 適用） |

**反例**：git 流程＋編碼規則塞同檔 → 想單獨重用編碼規則被綁住。
**口訣**：主題單一 + 能獨立理解 + 能單獨重用 → 就該獨立成檔（積木化）。

---

## 三、第二題：把 10 條混雜規則拆成 4 個子檔

爛範例（10 條全擠一起）：
```
1. Python 開頭加 PYTHONUTF8=1   2. commit 用繁中   3. 函式≤30 行
4. Next.js+Tailwind   5. push 前跑測試   6. SQL 參數化
7. Neon PostgreSQL   8. 不硬編碼路徑   9. 金鑰放環境變數   10. 分支 feature/<任務>
```

拆解結果：

| 子檔 | 收哪幾條 | 主題 | 重用範圍 |
|------|---------|------|---------|
| `code-quality.md` | ③ ⑧ ① | 程式碼本身怎麼寫 | 通用 → 全域 |
| `git-workflow.md` | ② ⑤ ⑩ | git 操作規矩 | 通用 → 全域 |
| `security.md` | ⑥ ⑨ | 防攻擊／防外洩 | 通用 → 全域 |
| `tech-stack.md` | ④ ⑦ | 這專案選什麼技術 | 專案專屬 → 留專案 CLAUDE.md |

拆完主檔只剩：
```markdown
# 我的專案
@code-quality.md
@git-workflow.md
@security.md
@tech-stack.md
```

兩個關鍵判斷：
1. **灰色地帶**：① ⑧ 既是品質也算可攜性 → 放 code-quality（寫每個檔的習慣）；可攜規則多時獨立 `portability.md` 也合理。**沒有唯一解，看哪些規則一起成長。**
2. **重用範圍**：前三檔通用 → 全域 `instructions/`；tech-stack 專案專屬 → 留專案。

---

## 四、第三題（陷阱題）：模組化的 4 個代價

| 風險 | 問題 | 解法 |
|------|------|------|
| ① 過度拆分 | 30 個 3 行小檔 → 翻 30 檔比大檔還難找 | 一主題一檔就停，3~50 行舒服，3 行的併回去 |
| ② 衝突無仲裁 | A 說≤30、B 說≤50 → 不可預測 | 主檔明訂優先級（語言鐵律最高、專案蓋全域） |
| ③ **context 沒省到** | `@import` 載入時**全展開合併**；500 行拆 10 檔還是 500 行 | 模組化省**維護+重用**非 token；省 token 靠 **skill 按需載入** |
| ④ 間接層遮蔽全貌 | 主檔只剩 `@import`，要跳進子檔才知內容 | 每個 `@import` 後補一句註解 |

**什麼時候不該拆**：
1. 整份本來就短（<100 行）→ 硬拆只增檔案數。
2. 規則高度耦合常一起改 → 拆開每次得開好幾檔。
3. 以為拆了能省 context（錯）→ 別為這理由拆。

> 🎯 模組化的真正價值是**維護性 + 跨專案重用，不是省 token**。
> 一主題一檔 + 優先序仲裁 + 主檔註解 = 好的模組化。
