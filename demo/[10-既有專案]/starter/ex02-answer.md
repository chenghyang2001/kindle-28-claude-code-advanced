# 練習 02 解答 — 把 review → simplify 串成固定流程

## 任務 1：串接順序與每步 gate

### 順序：先 `code-review`，後 `simplify`
口訣：**先弄對，再弄漂亮**。

| 外掛 | 處理什麼 | 性質 |
|------|---------|------|
| code-review | bug、安全漏洞（對不對） | 會出事，優先 |
| simplify | 冗餘、可讀性（好不好看） | 錦上添花，其次 |

**反過來（先 simplify）會怎樣：**
1. 白做工——先簡化得漂亮，review 發現邏輯要整段重寫，簡化全廢。
2. 更糟——simplify 會動手改，在有 bug 的程式上簡化，可能把 bug 藏更深。

### gate 條件（review → simplify 之間）
> `code-review` 列出的「🔴 必改項」全部修完、歸零，才能進 `simplify`。

「建議 / 備註」不是硬條件，列參考即可。

## 任務 2：完整流程（每步攔下/改善什麼）

```text
1) /code-review   → 收集必改項（攔下：injection 風險、邏輯錯誤）
2) 修掉必改項，確認歸零   ← gate，過了才往下
3) /simplify      → 對已正確的程式去冗餘、提可讀性（改善：拆長函式、去重複）
4) 人工複檢 diff   → 確認 simplify 沒改變行為
```

## 延伸思考：外掛何時幫倒忙 / 避免職責重疊
- **界線**：code-review 只准「找」不准「改」；simplify 才准「改」。守住這條，兩工具不會搶改同一段。
- **simplify 幫倒忙的典型**：在還有 bug 時就提早叫它——gate 條件正是用來防這件事。

## 關鍵收穫
- 正確性優先於美觀；simplify 一定排在 review 通過之後。
- 用「必改歸零」當硬 gate，串接才不會在錯的地基上美化。
