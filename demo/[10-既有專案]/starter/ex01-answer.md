# 練習 01 解答 — 啟用一個官方外掛並實跑

## 任務 1：`/code-review` 的用途與適用時機

### 用途
`/code-review` 是一個**自動程式碼審查員**。它讀取本次改動（diff），挑出三類問題：

- 🔴 **必改**：bug、邏輯錯誤、安全漏洞（SQL injection、金鑰外洩等）
- 🟡 **建議**：可讀性、命名、效能等改進空間
- 🟢 **備註**：純資訊提醒

它**只找問題、給建議，不動手改**——改不改、怎麼改由人決定。
（對比：`/simplify` 會真的動手改程式碼。）

### 適用時機
| 時機 | 原因 |
|------|------|
| 送 PR 之前 | 最標準，先自攔明顯問題 |
| 改動 ≥ 2 檔 或 ≥ 50 行 | 改越多人眼漏看率越高 |
| 高風險程式碼 | 使用者輸入 / DB 查詢 / 認證 / 金鑰 |

### 不該用的時機
- 改一個 typo、調一個設定值、只動 1～2 行 → 殺雞用牛刀。

## 任務 2：輸出整理（分類範例）

對某段改動跑 `/code-review` 後，把輸出整理成三類（非原樣貼上）：

| 分類 | 項目 | 處置 |
|------|------|------|
| 🔴 必改 | 使用者輸入未驗證（潛在 injection） | 立即修，gate 條件 |
| 🟡 建議 | 函式 > 30 行，可提取子函式 | 列入待辦 |
| 🟢 備註 | 變數命名可更具體 | 參考即可 |

## 關鍵收穫
- code-review = 隨叫隨到的資深同事，只圈問題不代筆。
- 「必改」是後續流程的硬 gate，「建議/備註」為參考。
