# 練習 01 解答 — 30 分鐘 onboard 陌生 codebase

## 核心心法：先畫地圖，再走路
最忌諱直接打開第一個檔從頭讀。先建高階全局地圖（有哪些區塊、怎麼串），要鑽細節再鑽。

## 任務 1：產出模組地圖與資料流（限只讀不改）

### 怎麼開始（由粗到細）
| 步驟 | 看什麼 | 為什麼 |
|------|--------|--------|
| 1. 目錄結構 | 資料夾怎麼分 | 結構透露架構（進入點/核心/設定/測試） |
| 2. `git log --oneline -20` | 最近改動 | 活躍的部分才是要先懂的 |
| 3. 進入點 | main / index / 啟動腳本 | 順資料流摸，比亂讀有方向 |

### 給 Claude 的指令範例
> 「不要改任何檔。先給我這個 repo 的高階架構：主要模組、進入點、資料流，
>  用一張文字圖呈現，並列出 3 個我該先讀的關鍵檔。」

輔助指令：
```bash
git log --oneline -20
git ls-files | sed 's@/[^/]*$@@' | sort -u | head
```

## 任務 2：標 3 個疑點追問到懂
- 對地圖中看不懂的模組／資料流，逐一追問（它為何存在、誰呼叫它、資料從哪來到哪去），直到能用一句話說清楚。

## ⚠️ 最關鍵紀律：只讀不改
onboard 階段唯一任務是「理解」不是「修改」。對專案還不懂時讓 AI 改東西很危險（改壞也看不出來）。先看懂，再動手。

> 🔗 呼應第九課決策表：onboard 用「探索 / code-review」這種只讀不改角色，不是 feature-dev。

## 關鍵收穫
- 先地圖後細節；用目錄 + git log + 進入點三步建立全局觀。
- 全程交代 Claude「只讀不改」，理解階段不動手。
