## 一、要解決的痛點：上下文腐敗（Context Rot）

AI 寫程式常見的崩潰劇本：**下午兩點像天才，下午四點開始鬼打牆**——忘記一小時前定好的架構規則、產出重複又無法編譯的廢碼。這不是 AI 變笨，而是**大腦超載**。

- 社群開發者 **Glitter Cowboy** 觀察到：Claude 處理 **30 分鐘內的短任務表現極佳**，但**數小時的長專案品質直線下降**，並把這病症命名為 **Context Rot（上下文腐敗）**。
- **比喻**：AI 的上下文視窗 = 它的「辦公桌 / 短期記憶」。長專案會在桌上堆滿失敗嘗試、除錯日誌、「抱歉我弄錯了」的廢棄對話，最後 AI 被雜訊淹沒，連真正的目標都找不到。
- 手冊圖表：**不用 GSD → 品質曲線一路溜滑梯、錯誤率暴增；用 GSD → 品質穩定維持高點。**

## 二、為什麼「開新對話」不夠？

手動清空、開新對話確實能清桌子，但等於餵 AI 一顆**失憶藥**：桌子乾淨了，但兩小時討論出的架構規範、技術限制也全忘了。每開一次就要重教一次，光貼背景知識就吃掉大半上下文。

## 三、GSD 的核心哲學：打碎重組 + 乾淨上下文

把龐大專案**拆成多個小階段**，且**每個階段都用全新乾淨的上下文執行**。透過嚴謹的**四階段循環**：

| 階段 | 指令 | 在做什麼 |
|---|---|---|
| 1. 討論 Discuss | `discuss-phase` | 不寫碼，反過來**問你問題**（技術堆疊、設計模式），先像跟架構師開會收斂規格 |
| 2. 計劃 Plan | `plan-phase` | 調查程式碼庫，寫出**極詳細計劃**：改哪些檔、順序、要補哪些測試 |
| 3. 執行 Execute | — | **關鍵**：在背景開一個 **100% 全新乾淨的 session**，只帶著那份計劃去幹活，從源頭消滅上下文腐敗 |
| 4. 驗證 Verify | — | 引導手動驗證 + 自動跑測試診斷 |

**接力賽比喻**：每一棒跑者（新 session）不需要知道前面跌倒幾次、討論多苦，只接過「計劃」這根接力棒，專心跑完自己的 100 公尺。

**失敗時的循環**：驗證失敗 → 讀錯誤訊息 → 結束當前執行 session → **帶著錯誤回到計劃階段重寫修復計劃** → 再開全新 session 改碼。**每次嘗試都是一張乾淨的辦公桌**，不會在同一視窗越改越糟。

## 四、記憶機制：把 AI 的大腦「外部化」

既然每次執行都是白紙，GSD 在專案根目錄建 **`.planning/` 資料夾**，裝滿 **Markdown 檔**當作 session 之間傳遞狀態的「記憶本」。背後的命令鏈工具會**自動把這些本地檔讀出來，注入新 session 的 system prompt**，所以新 AI 一上線「潛意識」就已載入專案狀態。

分工明確的狀態檔：

- **`PROJECT.md`** — 專案高階描述與重大決策
- **`REQUIREMENTS.md`** — 不能打破的軟硬性需求
- **`ROADMAP.md`** — 全局進度表（哪些等待中、哪些完成）
- **`STATE.md`** — 當下即時背景資訊
- **`PLAN.md`** — 正在執行的計劃

## 五、人類的新角色：審查者 + 指揮官

人類從「打字機」升級為**主管**。在 AI 寫完 `PLAN.md`、開新 session 狂奔**之前**，你有權審閱、用編輯器**直接手動修改那份 Markdown**。這正是 GSD 堅持「計劃」與「執行」分離的原因——防止 AI 幻覺把需求寫錯，導致後面所有接力跑者拿著錯誤接力棒衝下懸崖。

## 六、安裝與使用

- **安裝**：終端機 `npx get-shit-done-cc`（全域靜默安裝：`npx get-shit-done-cc --claude --global`）
- **全新專案**：`/gsd:new-project` → AI 像顧問般問一連串問題、自動生成狀態檔與完整 roadmap；隨時用 `progress` 查進度

## 七、進階武器

1. **既有老舊專案 → `map-codebase`**：不讓單一 AI 苦讀五年前的義大利麵程式碼，而是**同時派出多個探索者代理（Explorer Agents）並行偵查**（一個看資料庫連線、一個看使用者驗證、一個看前端…），彙整成系統地圖放在 `codebase/`，確保後續計劃貼合現有架構。

2. **多代理協作（並行執行）**：一個階段若含多個互不干涉的獨立任務（改資料庫 + 寫 API + 前端測試），可**同時啟動多個子代理，各自擁有專屬乾淨上下文**。用 `status` 像看儀表板一樣監控多個跑道。等於在本地端瞬間雇用架構師 + 前後端 + 測試工程師的完整團隊。

3. **⚠️ `--dangerously-skip-permissions`（危險略過權限）**：讓 GSD 進入**不間斷狂飆模式**（自己寫碼、自己跑測試、發現錯自己修，一路到完成），免去每次按 y 確認。手冊**最嚴厲警告：絕對只能用在安全的本地開發環境，嚴禁正式環境**——等於給虛擬團隊一把無限權限萬能鑰匙，理解錯就可能把運作中的資料庫清空。

## 八、結語的反思

GSD 本質是「**打碎重組 + 狀態檔案記憶**」的系統，用 Markdown 消滅上下文腐敗，並透過並行機制與探索者代理，把 Claude 提升為能**自主規劃、自主除錯、並行工作的專案大腦**。把原本卡關數小時的上下文管理，縮減到幾個指令。

最後拋出的思考題：**五年後人類的工作，還是打開編輯器寫程式，還是變成那個「按下新專案指令、喝著咖啡讀藍圖的指揮官」？我們究竟是在創造軟體，還是在管理一群不知疲倦的 AI 代理？**
