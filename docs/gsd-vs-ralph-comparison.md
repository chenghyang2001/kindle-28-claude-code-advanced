# GSD（第五課）vs Ralph 獨立版（第六課）— 詳細比較

> 建立日期：2026-06-22
> 用途：釐清兩個自動化開發框架的差別、使用時機與適用任務。

---

## 一、一句話定位

| 框架 | 一句話定位 |
|---|---|
| **GSD（Get Shit Done）** | 結構化的「**專案開發方法論**」：把一個專案從模糊想法，走到交付，全程有規劃 → 設計 → 執行 → 驗證的階段與品質關卡，人在每個關卡把關。 |
| **Ralph 獨立版** | 輕量的「**自主執行迴圈**」：給一份明確規格 + 完成信號，就無人值守地一直跑（寫 code → 跑測試 → 修）直到達標。 |

核心差異：**GSD 是「大腦」（幫你想清楚、分階段、把關品質）；Ralph 是「引擎」（埋頭把明確的苦工做完）。**

---

## 二、逐維度比較表

| 維度 | GSD（第五課） | Ralph 獨立版（第六課） |
|---|---|---|
| 核心理念 | 慢即是快：先想清楚再動手 | 放手讓它跑：目標明確就交出去 |
| 人類角色 | 決策者 / 審查者，全程 in-the-loop | 邊界守門員，設好目標就離開、事後驗收 |
| 自動化程度 | 半自動（多階段、多 sub-agent，但逐階段要你確認） | 全自動（fire-and-forget，可跑數小時） |
| 起點需要什麼 | 一個想法 / 需求（**可以模糊**）→ GSD 幫你問清楚 | 一份**清楚的規格** + **可自動驗證**的測試 / 完成條件 |
| 結束條件 | 階段目標達成 + 你審查通過 | 測試全綠 / 輸出完成暗號 / 撞 `--timeout` / `--calls` 上限 |
| 品質保證 | 內建 requirements → design → plan → verify → review 多重 gate + 多個 reviewer agent | 靠你**預先寫好的測試**當裁判（垃圾測試 → 垃圾結果） |
| 流程結構 | 重、明確分階段（discovery / roadmap / phase / execute / review） | 輕、單一迴圈（read → execute → test → repeat） |
| 監督方式 | 互動對話 + checkpoint 暫停問你 | tmux monitor 看即時 log |
| 主要風險 | 對小任務「殺雞用牛刀」、步驟多 | scope creep（自己加戲）、鬼打牆燒錢 → 需 rate limit / 斷路器 / timeout |
| 成本特性 | 你控制節奏，按階段花 | 可能長時間連續燒，必須靠護欄收斂 |
| 運行環境 | Claude Code 互動 session（本機即可） | headless `claude -p` + bash + tmux（Linux / WSL / VPS） |
| 典型輸出 | 一個有設計、有驗證紀錄的完整專案 | 一批「測試全綠」的明確產物 |

## 三、任務路由表（我有這種任務 → 該用哪個）

| # | 我的任務 | 該用 | 為什麼 |
|---|---|---|---|
| 1 | 我有個 SaaS 點子，想從零規劃並開發 | **GSD** | 需求模糊、要 roadmap / 設計 / 多階段 |
| 2 | 照這份 `api-spec.md` 實作 15 端點 + 測試，今晚跑完 | **Ralph** | 規格明確、可測試、要無人值守 |
| 3 | 幫既有大功能做架構設計與分階段計畫 | **GSD** | 以設計 / 規劃為主，要權衡 |
| 4 | 把這 30 個壞掉的單元測試修到全綠 | **Ralph** | 目標明確、機械、測試當裁判 |
| 5 | 重構整個模組，要考慮長期維護與權衡 | **GSD** | 要決策 / 審查 / 架構判斷 |
| 6 | 過夜把 100 個 lint error 清乾淨 | **Ralph** | 重複、可驗證、適合無人值守 |
| 7 | 需求還很模糊，需要有人幫我問清楚再做 | **GSD** | discovery / requirements gate |
| 8 | greenfield 小工具，規格清楚，想丟著自己長出來 | **Ralph** | 明確 + 可自動驗證 |
| 9 | 大專案要做 code review / security / 多維度驗證 | **GSD** | 多 reviewer agent + 品質 gate |
| 10 | CI 裡自動把失敗的 build 修到綠 | **Ralph** | `ralph-enable-ci`，可驗證、自動化 |

## 四、判斷口訣（30 秒選邊）

**選 GSD，如果：**

- 起點很模糊、需要先把需求 / 設計想清楚
- 任務跨多階段、需要架構決策與權衡
- 你想全程參與、在關卡上審查品質
- 正確性 / 可維護性比速度更重要

**選 Ralph 獨立版，如果：**

- 目標已經很明確（規格 / 測試已存在）
- 工作機械、重複、枯燥，但**可以自動驗證**
- 你想無人值守、丟著讓它跑（過夜 / CI）
- 你已寫好測試當客觀裁判，並能設好安全護欄

---

## 五、最佳組合（不是二選一）

兩者可以**串起來**：

> 先用 **GSD** 做規劃 + 設計 + 切出明確、可測試的 phase plan / spec，
> 再把其中「明確且可驗證」的執行階段，丟給 **Ralph 獨立版**無人值守跑。

**GSD 當大腦負責想清楚與把關，Ralph 當引擎負責把苦工做完。** 這樣既有結構與品質，又有自動化的效率。

---

## 六、一句話總結

- **GSD**：把「做對一個專案」這件事流程化、品質化。適合**從模糊到交付**的大局。
- **Ralph 獨立版**：把「做完一批明確苦工」這件事自動化、無人化。適合**明確且可驗證**的窄任務。
- 選錯的代價：用 GSD 做小苦工 = 殺雞用牛刀；用 Ralph 做模糊大專案 = 它會自己亂長、鬼打牆燒錢。
